# Big Bakes Tracker

**Big Bakes Tracker** is a live-only weekly attendance collection system for the six Big Bakes venues: Birmingham, Manchester, Liverpool, London East, London South, and Bake It Bristol. It queries the DesignMyNight availability API for the next 180 days, normalizes each standard session into a single event row, writes the current live snapshot into Airtable, derives weekly deltas from the latest two live snapshots, and sends a notification email summarizing the run.

The repository is intentionally scoped to **live data collection only**. Historical spreadsheet imports are out of scope and are not part of the deployed workflow. The system starts from the first live scrape on **2026-04-20**, and weekly delta creation is intended to begin from **Thursday 2026-04-23 onwards**, once a second live snapshot exists for comparison.

| Component | Purpose |
| --- | --- |
| `run_scraper.py` | Fetches live availability data for all six venues, normalizes the session rows, syncs Airtable Events and Snapshots, derives Weekly Deltas, and records a Run Log entry. |
| `send_notification_email.py` | Sends the production notification email using the latest run summary generated by the scraper. |
| `.github/workflows/weekly_scrape.yml` | Schedules the weekly Thursday run and supports manual execution for operational checks. |
| `requirements.txt` | Declares the minimal runtime dependency set for local runs and GitHub Actions. |

The attendance model uses a fixed reporting capacity of **24 seats** for every session. If the upstream API reports more than 24 attendees for a slot, stored **Tickets Sold** is capped at 24, **Tickets Remaining** is calculated as `24 - Tickets Sold`, and **Effective Attendance %** is calculated as `Tickets Sold / 24`. The raw API field is still preserved as **API Max Capacity** for review and anomaly detection.

| Weekly review rule | Trigger logic |
| --- | --- |
| Rule A | High-velocity sales on a previously slow-selling event close to the event date. |
| Rule B1 | API max capacity drops from positive to zero more than 21 days before the event, suggesting a likely private hire booking. |
| Rule B2 | API max capacity drops from positive to zero within 21 days of the event while previous period sales were low, suggesting a likely manual closure. |

Negative `Tickets Sold This Period` values are still stored in **Weekly Deltas** for auditability, but **Rule C is deprecated** and no longer raises a review flag or populates a review reason. This change was made after evaluation showed that negative deltas were predominantly routine refund and rebooking noise handled by other systems rather than actionable weekly review signals.

## Required environment variables

The scraper and notification workflow expect the following environment variables to be present.

| Variable | Purpose |
| --- | --- |
| `AIRTABLE_BASE_ID` | Airtable base identifier for the production tracker base. |
| `AIRTABLE_PAT` | Airtable personal access token with record read/write access. |
| `RESEND_API_KEY` | Resend API key used for notification email delivery. |
| `NOTIFICATION_EMAIL` | Recipient address for operational summaries and alerts. |

## Local execution

Install dependencies and run the scraper from the repository root.

```bash
python -m pip install --upgrade pip
pip install -r requirements.txt
export AIRTABLE_BASE_ID=...
export AIRTABLE_PAT=...
python run_scraper.py
```

To send a test notification with the most recent summary file, run:

```bash
export RESEND_API_KEY=...
export NOTIFICATION_EMAIL=...
python send_notification_email.py --mode test
```

## Scheduled execution

The GitHub Actions workflow runs every **Thursday at 07:00 UTC**, which corresponds to **08:00 Europe/London during British Summer Time** on the requested launch date. It also supports manual execution through the repository Actions interface.

## Operational notes

The workflow writes a run summary to `last_run_summary.json` in the repository workspace during execution. That summary is then reused by the notification script so the email reflects the exact production run outcome. Airtable remains the system of record for Venues, Events, Snapshots, Weekly Deltas, and the Run Log.
